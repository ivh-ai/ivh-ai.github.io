How it was built

Building Career HQ

A job-search operating system where markdown files are the database, a zero-dependency Node server is the dashboard, and custom Claude skills are the workforce. The architecture is the interesting part.

  • Node.js
  • Claude Skills
  • MCP
  • Markdown
  • Vanilla JS

The idea — and its three rules

A serious job search generates chaos: resume versions, dozens of postings, statuses, notes, research. Career HQ turns that into a system governed by three rules that shaped every technical decision:

  • One source of truth — every piece of career data is a plain, portable file
  • Log everything — untracked actions are lost actions
  • Never fabricate — the AI sharpens framing but invents no metrics, ever

Under the hood

Markdown as database

The data layer is a structured directory of markdown files: a tracker with a pipeline table, one file per application (status fields, fit assessment, a timestamped timeline log), and a profile folder. Structure is enforced by convention plus parsers, not a schema — every record stays human-readable, diffable, and version-controllable, and would still be useful if all the software around it vanished.

A dashboard in one file, no dependencies

The web dashboard is a single Node.js server file using only the standard library — no Express, no npm installs. It parses the markdown into JSON for a vanilla-JS front end — a two-tab interface with an at-a-glance overview and a drag-and-drop kanban board — and writes edits (move a card, log a touch, record an interview round) straight back into the markdown. Because the parsers and file conventions are coupled, the conventions are documented as a contract: change a tracker column, update the parser.

Paste a URL, get a card

Adding a job takes a link, not a form. A small endpoint fetches the posting and reads its structured JobPosting metadata (the JSON-LD that Greenhouse, Lever, and most career sites embed for Google), falling back to OpenGraph and title tags, then pre-fills an editable card — company, role, location, salary, source. The fetch is guarded against private and internal addresses, and anything it can't parse degrades to quick manual entry.

Claude skills as encoded workflows

Seven custom skills (resume tailoring, job discovery, application tracking, interview prep, role comparison, industry analysis, onboarding) each encode a workflow's rules once — what files to read, what conventions to follow, what honesty constraints apply. Any future Claude session invoked on the project follows the same playbook. Job-board data arrives through MCP connectors (Indeed, ZipRecruiter), discovered dynamically and merged, with web search as fallback — and the system still works with local files only when nothing is connected.

Packaging a private system for public viewing

The demo on this site is the real dashboard's front end, unmodified, running in its built-in demo mode. A small baked-in script forces demo mode and intercepts the API calls with a fictional sample pipeline, and demo mode already simulates every edit in the browser — so you can drag cards, log rounds, and add jobs, yet nothing is ever saved. Same pixels, fictional data, zero private information.

Key design decisions

Plain text over apps

Markdown outlives every SaaS tool and is natively readable by both humans and AI — the ideal shared memory between a person and their assistant.

Zero dependencies, zero rot

The dashboard uses only Node's standard library, so there is no dependency tree to break, audit, or update. Local tools should be boring to maintain.

Skills over prompts

Instead of re-explaining the rules in every conversation, each workflow is written once as a skill. Consistency across sessions is the feature.

Honesty as a hard rule

"Never fabricate resume content" is written into every skill, not left to chance. An AI job-search tool is only useful if its output can be trusted.

The Claude Code process

Career HQ was built inside-out: the file conventions first, then the skills that write to them, then the dashboard that visualizes them — each layer tested against real job-search use before the next was added. When the dashboard was later refocused into a dedicated tracker — a redesign taken from spec to plan to test-driven build — the markdown contract held, so the skills kept working untouched. Its built-in demo mode, designed in from the start, is what makes publishing this portfolio demo a packaging exercise rather than a rebuild.
